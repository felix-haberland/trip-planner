# Feature Specification: Yearly Vacation Planner — Slots, Trip Options, and a Yearly Chatbot

**Feature Branch**: `007-yearly-vacation-planner`
**Created**: 2026-04-19
**Status**: Draft
**Input**: User description: "Add a yearly-planning layer on top of the existing trip planner: a Year Plan with slots (free-text intent + user-defined time windows), Trip Options that fill those slots (with the same Suggest/Shortlist/Exclude triage as destinations today), a yearly overview UX, and a yearly-scoped chatbot that reuses the existing chat infrastructure and reasons across slots. Model the chat/conversation infrastructure so the same stack supports three scopes — Trip, Year Plan, and future Trip Option — without duplication."

## Clarifications

### Session 2026-04-19

- Q: Slot granularity — fixed months, arbitrary date ranges, or both? → A: **Both, with month-level as the default.** A slot is defined by a `start_month` + `duration_weeks` (the common case: "a week in June"). Users who need precision can also set an explicit `start_date` + `end_date` pair; when both are present, explicit dates win. Duration in either mode is approximate — the slot is a *planning unit*, not a calendar booking.
- Q: Can slots overlap, and can one slot span a year boundary (late Dec → early Jan)? → A: **Overlapping slots are allowed but flagged.** The UI warns on overlap ("Slot 'Christmas' overlaps with 'Ski week'") but does not block; real-world planning often has tentative alternatives competing for the same window. Year-spanning slots (e.g., Dec 27 → Jan 4) are allowed and belong to the Year Plan that contains the slot's **start** date. A slot cannot be split across two Year Plans.
- Q: Does a Trip Option carry its own criteria (activity weights, duration, climate) or inherit from the slot? → A: **Inherit with override.** The slot carries baseline criteria (`activity_weights`, `climate_hint`, `duration_weeks`, `constraints` freetext). A Trip Option inherits those as defaults but may override any field — e.g., a slot with `{golf: 60, relaxation: 40}` can host a Trip Option with `{golf: 100}` (pure-golf variant) alongside another with `{hiking: 70, culture: 30}` (alternative framing). Overrides are explicit on the Trip Option record; fields not overridden read from the slot at query time.
- Q: How does the AI balance "spread activities across the year" vs "user said they want 3 golf trips"? → A: **User intent wins; diversity is a soft default.** When the Year Plan's `overall_intent` or a slot's `activity_weights` explicitly encodes repetition (e.g., `overall_intent: "three golf weeks + a ski trip"` or multiple slots with `activity_weights.golf >= 50`), the AI respects that literally. When no such signal is present and slots are activity-ambiguous, the AI prefers diversity across the year as a tie-breaker (e.g., avoid suggesting a second beach trip into an empty slot if the previous slot already has a beach Trip Option shortlisted). Diversity is never used to override an explicit user statement.
- Q: Year rollover — are past-year plans archived, editable, or read-only? → A: **Soft-locked read-only after year-end, explicitly unlockable.** When the server date is past the Year Plan's `year` + 12 months (i.e., all of 2026's plan becomes read-only on 2027-01-01), the Year Plan is marked `status: locked`. All create/update/delete endpoints on the plan and its slots/options refuse writes with a `403 {code: "year_locked"}`. The UI shows a banner with an "Unlock (history correction only)" button; clicking it flips `status` back to `active` and writes a `locked_overridden_at` timestamp for auditability. Past-year plans are never deleted automatically and never hidden — they remain browseable from a Year Plans list.
- Q: Does a committed Trip remove its option from the slot, or does the slot hold both "planned" and "considered" options side-by-side? → A: **Side-by-side, with one option marked `committed`.** When the user drills from a shortlisted Trip Option into the concrete Trip planning flow and the Trip is created, the originating Trip Option's `committed_trip_id` is set. That option is rendered prominently ("Planned: <Trip title>") at the top of the slot. Other Trip Options on the same slot remain visible below, in their current state (shortlisted / suggested / excluded), as "alternatives" — so the user can keep a backup in view without losing it. At most one Trip Option per slot may be `committed` at a time; committing a second one requires uncommitting the first (a single-click reversal that clears `committed_trip_id` without touching the underlying Trip).
- Q: Conversation-owner unification scope — refactor all conversations now, or add the new scope in parallel and migrate later? → A: **Refactor now.** Add a polymorphic `owner_type` + `owner_id` pair to `conversations` (values: `trip`, `year_plan`, `trip_option`), migrate existing `trip_id` data into `(owner_type='trip', owner_id=trip_id)` via a one-time data migration (no schema drop — the old `trip_id` column is retained as a read-only compatibility shadow for this feature and dropped in a later spec), and wire all three scopes (Trip today, Year Plan in this feature, Trip Option reserved for the upcoming per-option chatbot) through the same `Conversation` and `ConversationMessage` models. The refactor is scoped to this feature because building Year Plan chat on a parallel conversation table would create exactly the duplication the forward-looking signal warned against.
- Q: How does the yearly chatbot's `suggest_trip_option_for_review` tool identify the target slot when the AI reasons across multiple slots in one response? → A: **Explicit `slot_id` required, with cross-slot suggestions allowed.** The tool takes a mandatory `slot_id` parameter — the AI must pick the slot it's suggesting into. To suggest one Trip Option as a candidate for multiple slots (e.g., "a warm golf trip would fit either June or September"), the AI calls the tool once per slot with the same `title` + `destination_hint`; the backend deduplicates by `(year_plan_id, title_norm)` and creates one Trip Option record linked to multiple slots via a `slot_trip_options` join table. Slot-option linking is many-to-many by design, matching the user's framing.
- Q: Activity-weights vocabulary — open dict, closed enum, or hybrid? → A: **Closed enum, extended.** The feature's own user stories include "ski week", "maybe-safari", and "Iceland aurora", none of which fit spec 006's original 10 tags. This spec extends the enumerated vocabulary to 13 by adding `skiing`, `safari`, and `aurora` alongside `golf`, `hiking`, `beach`, `city`, `culture`, `relaxation`, `food`, `nature`, `wellness`, `adventure`. Vocabulary remains closed (typos and drift are still rejected) and filters/sorts still key off the enum. Spec 006 FR-017a MUST be amended in the same PR to match so the two specs stay consistent.
- Q: Yearly chatbot failure handling when a single turn emits many tool calls across slots — what happens on partial failure? → A: **Keep-and-surface (no rollback, no auto-retry).** Suggestions that successfully persisted before the failure are kept (they are legitimate AI output the user can still triage). The assistant message MUST display a non-blocking error banner stating the error class (`api_error` / `rate_limit` / `tool_error` / `timeout`), the count of suggestions that made it through vs. the apparent intended total (best-effort from the partial tool-use trace), and a prompt to say "continue" to re-run. The backend MUST NOT auto-retry — that matches spec 006's no-auto-retry stance and keeps cost predictable. User agency (constitution III) is preserved: the user decides whether partial results are good enough or a re-prompt is needed.
- Q: Year-rollover "session" semantics — how is the unlock override scoped? → A: **Server-uptime check, no session infrastructure.** The override is valid iff `locked_overridden_at > server_start_time` (the uvicorn process's boot timestamp, captured on startup and held in memory). Server restart — including `uvicorn --reload` picking up a code change — re-applies the lock. Closing and reopening a browser tab does NOT by itself re-lock (the server is still the same uptime window); the user must explicitly unlock again only after a server restart. This keeps the implementation trivial (no cookies, no session table, no JWT) and matches constitution VI's simple-by-default stance. The worked example in US7.5 is updated to reflect "server restart" rather than "tab close" as the re-lock trigger.
- Q: Is the separately-proposed `list_slots_and_options` read tool redundant with the system prompt (which already carries slots + options each turn)? → A: **Drop the tool.** The system prompt (FR-011) is the single source of truth for slot + Trip Option state, rebuilt fresh on every turn. For the expected scale (≤15 slots, ≤50 Trip Options per Year Plan — well within Sonnet 4's context budget), a redundant tool (a) wastes tokens by re-serializing state the AI already has, and (b) risks prompt↔tool drift when the user triages mid-turn. The yearly chatbot's read-tool surface is therefore just `search_destinations` + `search_golf_resorts` / `search_golf_courses` — no new yearly-specific read tool. If scale grows beyond the context budget in a future release, a tool can be added then; it's a reversible decision.
- Q: How does a Trip Option's `vacationmap_region_key` get populated — does the AI pre-resolve, or does the backend auto-resolve from `destination_hint`? → A: **Backend auto-resolves via the existing spec 004 fuzzy resolver.** The `suggest_trip_option_for_review` handler calls `_resolve_lookup_key` on the AI-supplied `destination_hint` (and any optional `country_code` / `region_name` structured fields if the AI provides them) and fills `vacationmap_region_key` when it resolves. If ambiguous or unresolved, the key stays null and the tool's response includes `fuzzy_matched`, `matched_region`, and `other_regions_in_country` signals (mirroring spec 004's trip-scope response shape) so the AI can mention the resolution to the user. The user retroactively re-links unresolved options via the same autocomplete component already used for trip destinations. The AI is NOT required to pre-resolve; its job is to describe the destination in natural language, not to memorize VacationMap's region catalog.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Create a Year Plan with Slots and an Overall Intent (Priority: P1)

The user opens a new "Year Plans" section. They create a plan for 2026 and type a free-text overall intent: "One big adventure trip, two golf weeks, something warm in December, keep October open for a maybe-safari." They then lay out the slots they know they want to use — e.g., a week in March, 10 days in June, a week in September, two weeks in December (Christmas) — each with optional notes ("school holidays", "must be warm"). The plan is now the anchor for the rest of the flow.

**Why this priority**: Without a Year Plan and slots, nothing else in this feature can hang off anything. This is the unit the chatbot reasons over and the UX renders.

**Independent Test**: On a fresh install, the user creates a 2026 plan, adds three slots, writes an overall intent, and confirms the plan appears in the Year Plans list with its slots laid out on a timeline/grid.

**Acceptance Scenarios**:

1. **Given** the user is on the Year Plans list page, **When** they click "New Year Plan" and enter year `2026` and an `overall_intent`, **Then** a Year Plan is created with `status: active` and appears in the list.
2. **Given** a Year Plan exists for 2026, **When** the user adds a slot with `start_month: 6`, `duration_weeks: 1`, and label "Golf week", **Then** the slot is persisted and rendered in the yearly overview at the right month position.
3. **Given** a Year Plan exists, **When** the user adds a slot with explicit `start_date: 2026-12-27` and `end_date: 2027-01-04` (year-spanning), **Then** the slot is accepted, belongs to the 2026 Year Plan (based on `start_date`), and is rendered crossing the December–January boundary on the 2026 overview.
4. **Given** two slots overlap in time, **When** the user saves the second one, **Then** the save succeeds but the UI shows a non-blocking warning banner listing the overlap ("Slot 'Christmas' overlaps with 'Ski week' on Dec 27–Jan 4").
5. **Given** a slot exists, **When** the user edits its label, constraints, or `activity_weights`, **Then** the changes persist and are reflected in the yearly overview and in the next chatbot turn's system prompt.
6. **Given** a Year Plan is created for a year that already has a plan (2026 exists, user tries to create another 2026), **When** the user submits, **Then** the system allows multiple plans per year (they are distinguished by title/notes) — the feature explicitly supports parallel plans per year for "what-if" planning.

### User Story 2 - Yearly Chatbot Suggests Trip Options into Slots Across the Year (Priority: P1)

The user opens the Year Plan's chatbot and says: "Help me shape 2026. I've got slots in March, June, September, and December. I know I want two golf weeks and something warm for Christmas — be creative with the others." The AI reads the overall intent, the slots with their constraints, the user's profile, and the past trip + visit history (including `visit_again` annotations). It proposes Trip Options into the relevant slots — e.g., "Algarve golf for March, Andalusia golf for September, Maldives for Christmas, Iceland aurora for June as a wildcard" — and reasons across slots ("if June is Iceland then September staying warm makes sense"). Each suggestion enters the slot's Pending Review list. The user triages.

**Why this priority**: This is the headline capability. The whole point of a yearly layer is AI-assisted combinatorial reasoning across slots that per-trip planning cannot express.

**Independent Test**: With a 2026 Year Plan containing 4 slots and the user's existing visit history in `vacation.db`, ask the chatbot "plan my year". Verify that within one assistant turn, multiple `suggest_trip_option_for_review` tool calls are emitted targeting different slots, that none target slots marked `locked`, and that at least one suggestion's `ai_reasoning` references cross-slot trade-offs or past visits.

**Acceptance Scenarios**:

1. **Given** a Year Plan with 4 slots and a visit-history entry for `PT:Algarve` (March 2025, `visit_again: not_soon`), **When** the user asks for yearly suggestions, **Then** the AI does NOT suggest Algarve for the March slot and surfaces it in its response as "Algarve would have fit March but you visited recently".
2. **Given** a Year Plan with 4 slots, **When** the yearly chatbot responds, **Then** each `suggest_trip_option_for_review` call carries a concrete `slot_id`, a `title`, a `destination_hint` (region, country, or area — may be freetext), optional `activity_weights` override, optional `duration_weeks` override, and `ai_reasoning`.
3. **Given** the AI believes one Trip Option fits multiple slots (e.g., "warm golf could be June or September"), **When** it emits suggestions, **Then** it calls the tool once per slot with the same `title` and matching `destination_hint`; the backend creates one Trip Option and links it to both slots via the `slot_trip_options` join.
4. **Given** the Year Plan's `overall_intent` says "three golf weeks", **When** the chatbot suggests options, **Then** at least three suggestions are golf-weighted (explicit `activity_weights.golf >= 50` on the option or the slot). The AI does NOT "diversify away" explicit user intent.
5. **Given** the chatbot reasoning references visit-history patterns (e.g., "avoiding SE Asia after Thailand last year"), **When** the user inspects any suggested Trip Option's `ai_reasoning`, **Then** such pattern notes are visible inline on the option card.
6. **Given** a slot has `status: locked` (e.g., already committed to a trip the user wants to leave alone), **When** the AI responds, **Then** it does NOT emit `suggest_trip_option_for_review` for that slot; it may still reference the committed trip as context.

### User Story 3 - Triage Trip Options (Suggested → Shortlisted / Excluded) (Priority: P1)

The chatbot has populated the slots with suggestions. The user opens the yearly overview, sees the Pending Review count on each slot, and clicks through each option. Shortlist, exclude (with a reason), unreview, reconsider — the same lifecycle as destination triage today, applied to Trip Options at the yearly scope. Nothing moves to Shortlisted or Excluded without an explicit user click.

**Why this priority**: This is the codification of constitution principle III ("The User Owns the Decisions") at the yearly scope. It's how the user asserts control over the AI's suggestions.

**Independent Test**: Generate 5 yearly Trip Option suggestions. Shortlist 2, exclude 2 (with reasons), leave 1 in Pending. Refresh. Verify each ends up in the expected bucket on its slot(s).

**Acceptance Scenarios**:

1. **Given** a Trip Option is `status: suggested` on a slot, **When** the user clicks Shortlist (optional note), **Then** the option's status becomes `shortlisted`; `ai_reasoning`, `destination_hint`, and any AI-derived fields are preserved.
2. **Given** a Trip Option is `status: suggested`, **When** the user clicks Exclude with a reason, **Then** the status becomes `excluded` and the reason is stored; if the AI set a `pre_filled_exclude_reason` on the suggestion (e.g., "overlaps with visit to nearby region last year"), it is pre-populated in the exclude prompt.
3. **Given** a Trip Option is shortlisted, **When** the user clicks Unreview, **Then** it returns to `suggested`; when they click Exclude, **Then** it moves to `excluded`.
4. **Given** a Trip Option is excluded, **When** the user clicks Reconsider (optional new note), **Then** it moves to `shortlisted`; the original exclusion reason is dropped.
5. **Given** a Trip Option is linked to multiple slots, **When** the user triages on any one slot, **Then** the option's status changes globally (it's one record, shared across its slots). The UI makes this clear ("Excluding this option will affect slots June and September").
6. **Given** the chatbot calls `suggest_trip_option_for_review` with a `title` or `destination_hint` that already exists on a shortlisted or excluded option on the same Year Plan, **When** the tool runs, **Then** it returns `{status: "rejected", reason: "Already in <list> list on slot(s) X"}` and no new record is created.

### User Story 4 - Drill Into a Shortlisted Trip Option to Create the Concrete Trip (Priority: P1)

The user has a shortlisted option "Golf week — Algarve, first week of June". They click "Plan this trip". The existing trip-planning flow opens, pre-filled with the option's data: title, month/duration, `activity_weights`, `climate_hint`, `destination_hint` as a seed suggestion. They continue in the existing per-trip chatbot and shortlist actual destinations. When the Trip is saved, it is linked back to both the slot and the Trip Option — the Year Plan overview now shows "Planned: <Trip title>" at the top of that slot, with the other options still visible as alternatives below.

**Why this priority**: This is the bridge between the yearly layer and the existing trip planner. Without it, the yearly plan is a dead-end whiteboard; with it, the existing tool becomes the per-slot detail view.

**Independent Test**: Shortlist a Trip Option, click "Plan this trip", verify the existing trip planner opens pre-filled, save a Trip, return to the Year Plan, verify the slot shows the trip as "Planned" with a link.

**Acceptance Scenarios**:

1. **Given** a shortlisted Trip Option, **When** the user clicks "Plan this trip", **Then** the existing trip creation flow opens with fields pre-filled from the option (`title`, approximate month/duration, `activity_weights`, `climate_hint`, `destination_hint`).
2. **Given** the user completes trip creation, **When** the Trip is persisted, **Then** the Trip Option's `committed_trip_id` is set to the new Trip's id and the slot renders "Planned: <Trip title>" at the top.
3. **Given** a Trip Option is committed, **When** another option on the same slot is committed, **Then** the save is refused and the UI prompts "<First option title> is already committed on this slot. Uncommit first?" with a single-click Uncommit action.
4. **Given** a Trip Option is committed, **When** the user uncommits, **Then** `committed_trip_id` is cleared; the linked Trip is NOT deleted (it remains in the trip planner untouched), only the slot-linkage is severed. The Trip Option returns to `shortlisted`.
5. **Given** a Trip Option is linked to multiple slots and becomes committed on one of them, **When** the other slots render, **Then** they continue to show the option in its `shortlisted` state with an informational badge "Committed as Trip on another slot", so the user knows it's already taken without removing it from their list of considered alternatives.
6. **Given** the linked Trip is deleted via the existing trip planner, **When** the yearly overview refreshes, **Then** the Trip Option's `committed_trip_id` is cleared (dangling FK is cleaned by the application) and the slot reverts to showing the option as `shortlisted`.

### User Story 5 - Yearly Overview UX (Timeline + Summary Cards) (Priority: P2)

The Year Plan page renders the year as a horizontal timeline spanning Jan → Dec, with slots laid out at their time position and sized by duration. Each slot is a card showing: its label, dates/months, Pending/Shortlisted/Excluded counts, the committed Trip (if any), and an inline list of up to 3 top shortlisted options. Clicking a slot opens a drawer with the full list of Trip Options for that slot. A month-grid fallback is available for narrow viewports.

**Why this priority**: Without a coherent visual, the yearly-reasoning value doesn't land — users need to *see* the year shape to judge trade-offs.

**Independent Test**: With a Year Plan that has 4 slots across the year and ~10 Trip Options distributed across them, open the overview on a wide screen, confirm the timeline renders slots at their correct positions with their option counts, and confirm clicking any slot opens a drawer showing that slot's options grouped by status.

**Acceptance Scenarios**:

1. **Given** a Year Plan with slots spanning multiple months, **When** the overview renders on a wide viewport (>1000px), **Then** the timeline shows Jan–Dec axes and each slot is positioned at its start date/month with width proportional to its duration.
2. **Given** a slot has 3 shortlisted and 2 pending options, **When** the slot card is rendered, **Then** the counts are visible at a glance and the top 3 shortlisted options are listed inline.
3. **Given** a slot has a committed Trip, **When** the slot card is rendered, **Then** "Planned: <Trip title>" appears at the top of the card with a link that opens the Trip in the existing trip planner.
4. **Given** the viewport is narrow (<700px), **When** the overview renders, **Then** it falls back to a month-grid layout (one row per month) without losing functionality.
5. **Given** the user clicks a slot card, **When** the drawer opens, **Then** it shows every Trip Option for that slot grouped by status (Shortlisted / Pending / Excluded) with triage buttons.
6. **Given** two slots overlap in time, **When** they render, **Then** they are visually stacked (not hidden) and the overlap is visually indicated (e.g., offset or border accent).

### User Story 6 - Unified Conversation-Owner Refactor (Priority: P2)

Today, `conversations` are scoped to a trip (`trip_id`). This feature refactors that to a polymorphic `(owner_type, owner_id)` pair so the same `Conversation` and `ConversationMessage` tables power chat at three scopes: Trip (today), Year Plan (this feature), and Trip Option (reserved for a planned follow-up feature — no UI in this spec). The yearly chatbot in US2 runs on the new unified shape from day one.

**Why this priority**: Without the refactor, we'd either build a parallel yearly-conversations table (duplication) or shoehorn Year Plan chat into `trip_id` (wrong semantics). Both approaches were flagged in the feature request as the thing to avoid. Doing it now, while there's only one existing scope, is strictly cheaper than doing it later when there are three.

**Independent Test**: After the feature ships, (a) open a Trip — its existing conversations still load and new messages post correctly; (b) open a Year Plan — a new "Main" conversation is auto-created and yearly-scoped chat works; (c) run a data-integrity check confirming every `conversations` row has a populated `(owner_type, owner_id)` pair and that `trip_id` (if retained as shadow) matches for `owner_type = 'trip'` rows.

**Acceptance Scenarios**:

1. **Given** the feature ships with a one-time data migration, **When** the app starts, **Then** every existing `conversations` row has `owner_type = 'trip'` and `owner_id` equal to the row's original `trip_id`.
2. **Given** a new Year Plan is created, **When** the user opens its chatbot for the first time, **Then** a "Main" conversation is auto-created with `owner_type = 'year_plan'` and `owner_id = <year_plan_id>`, mirroring the auto-create-on-first-open behavior that trips have today (spec 003 FR-002).
3. **Given** the user sends a message in a Year Plan conversation, **When** the backend processes it, **Then** the chat endpoint is scoped to the conversation (`/api/conversations/{conv_id}/messages`), not to the year plan — matching the existing conversation-scoped endpoint shape (spec 003 FR-007).
4. **Given** a Year Plan has multiple conversations ("Main", "October alternatives"), **When** the user switches between them, **Then** the chat history reloads but the year state (slots, options) stays the same — same separation of concerns as trips today (spec 003 FR-003).
5. **Given** a Year Plan is deleted (cascade per constitution principle I requires user confirmation), **When** deletion completes, **Then** its conversations and messages are deleted but no other Year Plan or Trip is affected.
6. **Given** a future feature adds Trip Option chat (`owner_type = 'trip_option'`), **When** that feature ships, **Then** it requires zero changes to the `conversations` / `conversation_messages` schema — only a new value in the `owner_type` enum and a new system-prompt assembler.

### User Story 7 - Past-Year Plans Are Soft-Locked After Rollover (Priority: P3)

On January 1st of the year after the plan's year, the Year Plan auto-locks (status: `locked`). Writes (new slots, edits, triage) are refused with a clear "This year plan is locked" message. A banner offers an explicit Unlock button for history correction; clicking it reverts status to `active` and records `locked_overridden_at`. The plan is never deleted, never hidden.

**Why this priority**: Keeps historical plans intact without user effort (the point is accidental edits don't quietly rewrite history), while preserving the escape hatch. Ships last because it only matters at year-end and has minimal day-to-day value.

**Independent Test**: Set the system clock forward past a Year Plan's year-end, reload, confirm writes return `403 {code: "year_locked"}` and the UI banner shows. Click Unlock, confirm a subsequent write succeeds.

**Acceptance Scenarios**:

1. **Given** a Year Plan for 2026 and the current date is 2027-01-01 or later, **When** the user opens the plan, **Then** its `status` is `locked` (computed or stored) and the UI shows a "Locked — history correction only" banner.
2. **Given** a Year Plan is locked, **When** the user attempts to add a slot, edit an option, triage, or send a chatbot message, **Then** the backend returns `403` with `{code: "year_locked"}` and the UI surfaces a friendly "Unlock to correct history" prompt.
3. **Given** a Year Plan is locked, **When** the user clicks Unlock and confirms, **Then** `status` flips to `active`, `locked_overridden_at` is set, and subsequent writes succeed.
4. **Given** a Year Plan was unlocked, **When** the user views it, **Then** the UI subtly indicates "Unlocked for history correction on <date>" so future-them knows why the past was touched.
5. **Given** a 2026 plan had `locked_overridden_at` set during the current server process, **When** the server restarts (including on code-reload via `uvicorn --reload`), **Then** the plan re-locks automatically because `locked_overridden_at` is no longer greater than the new `server_start_time`. Closing a browser tab alone does NOT re-lock — the server uptime is what gates the override.

### Edge Cases

- **Empty Year Plan**: A plan with no slots is valid — the user may start with just an `overall_intent` and let the AI propose slots too (a future enhancement; v1 treats an empty plan as a valid but non-actionable state; the chatbot responds with "add at least one slot so I can suggest options").
- **Deleting a slot that has Trip Options**: The delete is refused if any option is `shortlisted` or `committed`. `suggested` and `excluded` options on a deleted slot are detached from that slot (if they have other slots) or cascade-deleted if they were only linked to the deleted slot.
- **Deleting a Year Plan**: Cascade-deletes its slots, its conversations (per existing conversation cascade rules), and its Trip Options. Committed Trips are NOT deleted — only their back-links to Trip Options are severed. Requires explicit user confirmation (constitution principle I).
- **Trip Option without a slot**: Not allowed at create time; `slot_trip_options` requires at least one slot link. Removing the last slot link of an option soft-archives the option (status: `orphaned`) rather than losing the AI reasoning.
- **Chatbot asked about a locked Year Plan**: The chatbot refuses to send messages (`403` same as other writes) but the user can still *read* existing message history. An explicit read-only mode preserves historical context.
- **Overlapping slots and committed Trip**: If slot A has a committed Trip and slot B overlaps slot A and also has suggestions, the AI includes in its system-prompt context the committed Trip's destination + dates so it avoids suggesting a conflicting option ("you're already going to Algarve that week").
- **Visit history reads through to the yearly layer**: Same 3-tier semantics as spec 005 apply: `never` / `not_soon` hard-filter from the yearly chatbot's internal `search_destinations` calls; `few_years` annotated; `anytime` included. The yearly chatbot MUST also read each shortlisted/committed Trip Option's `destination_hint` as "already taken" context so it doesn't re-suggest the same region into a different slot within the same year unless the user's intent explicitly asks for it.
- **Same destination across two slots in the same year**: Allowed (user might want Portugal in March and Portugal in October for different activities), but the AI must surface it explicitly in its reasoning ("I'm proposing Portugal again for October — you said you'd enjoy a return for wine harvest").
- **Explicit override of year boundary**: If the user moves a December slot's `end_date` into January of the next year after the next-year plan exists, the slot stays on the original plan (start-date wins) and an informational note appears on both plans about the overlap.

## Requirements *(mandatory)*

### Functional Requirements

#### Year Plan + Slots data model

- **FR-001**: System MUST persist Year Plans in `trips.db` with attributes: `id`, `year` (int, e.g., 2026), `title` (nullable — optional human label for disambiguating parallel plans per year), `overall_intent` (markdown, nullable), `status` (`active` / `locked`), `locked_overridden_at` (nullable timestamp), `created_at`, `updated_at`. Multiple Year Plans per `year` are allowed.
- **FR-002**: System MUST persist Slots in `trips.db` with attributes: `id`, `year_plan_id` (FK), `label` (required, e.g., "March", "Christmas", "Golf Week 1"), `start_month` (1–12, nullable when explicit dates are set), `duration_weeks` (nullable — default planning unit), `start_date` (nullable ISO date), `end_date` (nullable ISO date), `activity_weights` (JSON dict keyed by the extended closed vocabulary defined in FR-002d — empty dict means no baseline), `climate_hint` (nullable freetext, e.g., "warm", "any"), `constraints` (nullable freetext, e.g., "school holidays", "avoid red-eyes"), `display_order` (int, for UI stability), `status` (`open` / `locked` — `locked` is per-slot, e.g., when the user wants the chatbot to stop suggesting into a slot), `created_at`, `updated_at`.
- **FR-002a** (slot granularity): When both `(start_month, duration_weeks)` and `(start_date, end_date)` are present on a slot, the explicit dates take precedence for layout and chatbot reasoning; month/duration are retained as a fallback. At least one of the two modes MUST be populated at save time.
- **FR-002b** (year-boundary slots): A slot with `start_date` in year Y and `end_date` in year Y+1 is permitted. The slot belongs to the Year Plan whose `year` equals `start_date.year`. The UI on the Y+1 plan MAY display an informational note about the spill-over but does not own the slot.
- **FR-002c** (overlap policy): Slot overlap validation is advisory only — the backend MUST NOT reject a slot create/update solely because it overlaps another. The API response MUST include a `warnings` array enumerating overlaps so the UI can surface them non-blocking.
- **FR-002d** (activity-weights vocabulary — extended closed enum): `activity_weights` on Slots and on Trip Options (FR-003) MUST use the extended closed vocabulary: `golf`, `hiking`, `beach`, `city`, `culture`, `relaxation`, `food`, `nature`, `wellness`, `adventure`, `skiing`, `safari`, `aurora` (13 tags). Unknown keys MUST be rejected at write time with a 400 error listing the offending keys. Values remain integers 0–100 (not strictly summed to 100). Spec 006 FR-017a (trip-level `activity_weights`) MUST be amended in the same PR to reference the same 13-tag vocabulary so per-trip, per-slot, and per-option weights remain interchangeable. The vocabulary is intentionally kept small and user-recognizable — further additions require a spec amendment, not silent drift.

#### Trip Option data model + slot linkage

- **FR-003**: System MUST persist Trip Options in `trips.db` with attributes: `id`, `year_plan_id` (FK), `title` (required), `title_norm` (derived — same normalization as spec 006 FR-003a: lowercase + collapse whitespace + strip ASCII punctuation), `destination_hint` (freetext — region, country, or area; nullable but recommended), `vacationmap_region_key` (nullable link in `country_code:region_name` form, same stable-id pattern as constitution IV), `activity_weights_override` (JSON dict keyed by the extended closed vocabulary in FR-002d, nullable — when null, read from slot), `climate_hint_override` (nullable), `duration_weeks_override` (nullable), `ai_reasoning` (markdown — AI's rationale, preserved across triage transitions), `scores_snapshot` (JSON, nullable — snapshot from any `search_destinations` call the AI made), `status` (`suggested` / `shortlisted` / `excluded` / `orphaned`), `user_note` (markdown, nullable — captured on shortlist or reconsider), `excluded_reason` (nullable), `pre_filled_exclude_reason` (nullable — AI-suggested exclusion hint, same pattern as spec 002 FR-007), `committed_trip_id` (nullable FK to `trips` — set when drilled into the trip planner), `created_at`, `updated_at`.
- **FR-003a** (many-to-many slot link): System MUST persist a `slot_trip_options` join table with attributes: `slot_id` (FK), `trip_option_id` (FK), `display_order` (int — per-slot ordering), `created_at`. Primary key is the pair. A Trip Option MUST have at least one slot link at create time; removing the last link transitions the option to `status: orphaned` rather than deleting it.
- **FR-003b** (committed-exclusivity): At most one Trip Option per slot MAY have `committed_trip_id` set at a time. The backend MUST enforce this at commit time by refusing a second commit with a `409 {code: "slot_already_committed", committed_option_id: ...}` response.
- **FR-003c** (resolved criteria read): When the chatbot or UI needs the *effective* criteria for a Trip Option (i.e., the option's overrides layered on top of its slots' baselines), the system MUST resolve as follows: for each field (`activity_weights`, `climate_hint`, `duration_weeks`), if the option has a non-null override, use it; otherwise, use the slot's value. When the option is linked to multiple slots with different slot-level values, the UI MUST show the per-slot resolution; the chatbot MUST receive both per-slot baselines and the option's overrides so it can reason about the mismatch.

#### Triage lifecycle (reuse of spec 002 pattern)

- **FR-004**: System MUST expose exactly one state-mutating tool to the yearly chatbot for Trip Options: `suggest_trip_option_for_review`. All other yearly tools MUST be read-only. This matches spec 002 FR-001.
- **FR-004a**: `suggest_trip_option_for_review` MUST accept: `slot_id` (required), `title` (required), `destination_hint` (optional), `vacationmap_region_key` (optional), `activity_weights_override` (optional), `climate_hint_override` (optional), `duration_weeks_override` (optional), `ai_reasoning` (required — the AI's rationale), `scores_snapshot` (optional), `pre_filled_exclude_reason` (optional).
- **FR-004b** (cross-slot dedup): When the AI calls `suggest_trip_option_for_review` multiple times in the same turn with matching `(year_plan_id, title_norm)`, the backend MUST create exactly one Trip Option record and link it to all referenced slots via `slot_trip_options`. Subsequent calls for already-linked `(slot_id, trip_option_id)` pairs return `{status: "already_linked"}` without side-effects.
- **FR-004c** (region auto-resolution): The `suggest_trip_option_for_review` handler MUST invoke spec 004's fuzzy region resolver (`_resolve_lookup_key`) on the AI-supplied `destination_hint` (plus any structured `country_code` / `region_name` hints the AI also passes). If the resolver returns a stable `country_code:region_name` key, the handler populates `vacationmap_region_key` on the new Trip Option. If it does not resolve, `vacationmap_region_key` stays null. In both cases the tool response MUST include spec-004-style signals `fuzzy_matched` (bool), `matched_region` (object, when matched), and `other_regions_in_country` (list, when the country has alternatives) so the AI can explain the resolution to the user in the same turn. The AI MUST NOT be required to pre-resolve region keys. When `vacationmap_region_key` is null, the UI surfaces a "Link to VacationMap region" autocomplete (the same component used for trip-scoped manual linking) so the user can link retroactively.
- **FR-005**: System MUST provide UI actions (and backing HTTP endpoints) for: Suggested → Shortlisted, Suggested → Excluded, Shortlisted → Excluded, Shortlisted → Suggested (Unreview), Excluded → Shortlisted (Reconsider) — mirroring spec 002 FR-003.
- **FR-005a**: All triage transitions MUST be triggered by explicit user action (HTTP POST), never by AI tool calls — constitution III.
- **FR-006**: `suggest_trip_option_for_review` MUST reject (without inserting) a Trip Option whose `(year_plan_id, title_norm)` matches an existing `shortlisted` or `excluded` option on the same Year Plan, returning `{status: "rejected", reason: "Already in <list> list on slot(s) X, Y"}`. Matching `suggested` options on a *different slot* are not rejected — the cross-slot-dedup rule (FR-004b) handles those by linking instead of duplicating.
- **FR-007**: When moving Suggested → Shortlisted, the system MUST preserve `ai_reasoning`, `destination_hint`, `vacationmap_region_key`, and `scores_snapshot` — mirroring spec 002 FR-008.
- **FR-008**: When moving Suggested → Excluded, the system MUST preserve `ai_reasoning` and require a non-empty `excluded_reason` — mirroring spec 002 FR-009.
- **FR-009**: When moving Excluded → Shortlisted (Reconsider), the system MUST drop `excluded_reason` but allow a new optional `user_note` — mirroring spec 002 FR-010.
- **FR-009a** (shared status across slots): A Trip Option's `status` is a single field on the option record, not per-slot. Triaging from any slot card affects the option globally. The UI MUST surface this clearly when the option is linked to multiple slots (e.g., "Excluding this option will remove it from slots June and September").

#### Yearly chatbot (scope = Year Plan) and unified conversation owner

- **FR-010** (unified conversation owner — refactor): System MUST extend the `conversations` table with columns `owner_type` (enum: `trip`, `year_plan`, `trip_option`) and `owner_id` (int). Existing rows MUST be migrated in a one-time data fill: `owner_type = 'trip'`, `owner_id = trip_id`. The existing `trip_id` column is retained as a read-only compatibility shadow for this feature and MUST be dropped in a future spec (see Future Work).
- **FR-010a** (conversation endpoints): The existing conversation-scoped endpoint shape (`/api/conversations/{conv_id}/messages`) is unchanged — spec 003 FR-007 already ensured scoping is at the conversation, not the trip. New endpoints to *list/create* conversations under a Year Plan (`GET /api/year-plans/{id}/conversations`, `POST /api/year-plans/{id}/conversations`) MUST return the same `Conversation` shape as trip-scoped endpoints.
- **FR-010b** (auto-create Main): When a Year Plan is opened for the first time, the system MUST auto-create a "Main" conversation (`owner_type = 'year_plan'`, `owner_id = <year_plan_id>`, `name = "Main"`) — mirroring spec 003 FR-002 for trips.
- **FR-010c** (reserved: trip_option): The `owner_type` enum MUST include `trip_option` as a reserved value. No UI or tool in 007 creates such conversations; the enum value exists so the upcoming per-trip-option chatbot (future spec) is a pure additive — new system-prompt assembler only, no schema change.
- **FR-011** (system prompt assembly for Year Plan chat): On every message, the yearly chatbot system prompt MUST be assembled from: (1) `instructions.md` (same file as trip-chat, with a new "Yearly planning" section), (2) `profile.md`, (3) the current Year Plan state (overall_intent + all slots with their constraints and their Trip Options grouped by status), (4) visit history with `visit_again` annotations (same shape as spec 005 FR-005), (5) a summary of any other Year Plans for the same year and their committed Trips (so the chatbot doesn't silently clash between parallel plans), (6) a list of committed Trips already planned for the current plan (destination, dates). All six are read fresh on every message — constitution II.
- **FR-012** (yearly-scoped read tools): The yearly chatbot's read-tool surface MUST be exactly `search_destinations` (same as trip-chat), `search_golf_resorts`, and `search_golf_courses` (spec 006). No yearly-specific read tool is introduced. Slot and Trip Option state is authoritative in the system prompt (FR-011), rebuilt fresh every turn — the tool surface stays intentionally small. If future scale exceeds the context budget, a scoped read tool MAY be added via an additive spec; it is not in v1.
- **FR-012a** (activity-weights propagation): Same convention as spec 006 FR-017a — when a slot or Trip Option has `activity_weights.golf >= 50`, the chatbot treats the golf tools as the primary suggestion source for that slot/option. The system prompt extends spec 006's instructions to include slot-level and option-level weights.
- **FR-013** (instructions for cross-slot reasoning): `instructions.md` MUST be updated with a new "Yearly planning" section instructing the AI to: (a) read past visits and existing committed Trips before suggesting; (b) respect explicit user statements about repetition (e.g., "three golf weeks" → do not diversify away); (c) prefer geographic and activity diversity across slots as a tie-breaker only; (d) explicitly reference cross-slot trade-offs in its `ai_reasoning` when relevant ("if June is Iceland, September should be warm"); (e) never suggest into a locked slot or a locked Year Plan; (f) use the `name_query` convention from spec 006 when the user references a specific named entity.
- **FR-013a** (partial-failure handling in yearly turns): When a single yearly chat turn emits multiple `suggest_trip_option_for_review` tool calls and fails mid-loop (Anthropic API error, rate limit, timeout, or malformed tool result), the backend MUST: (a) commit every suggestion that successfully persisted before the failure point — no transactional rollback of the turn; (b) attach a structured error envelope to the assistant message with fields `{status, message, succeeded_count, attempted_total_estimate, slot_ids_populated}` where `status ∈ {api_error, rate_limit, tool_error, timeout}`; (c) surface the envelope as a non-blocking banner on the assistant message with a one-click "Continue" affordance that re-prompts the AI with the remaining slot context and already-suggested options in its system-prompt state. The backend MUST NOT auto-retry (matches spec 006 FR-008 stance). Tool calls that were in-flight when the failure occurred are not replayed silently; the next turn sees the accumulated state and decides what to add.

#### Commit Trip flow (bridge to existing per-trip planner)

- **FR-014**: System MUST provide an HTTP endpoint `POST /api/trip-options/{id}/commit` that, given a shortlisted Trip Option, (a) creates a new Trip via the existing trip-creation code path pre-filled with the option's `title`, resolved `activity_weights`, `climate_hint`, `duration_weeks`, start month/date from the option's first linked slot, and (b) sets the option's `committed_trip_id`. The endpoint returns the new Trip's id so the frontend can navigate into the existing trip planner.
- **FR-014a** (pre-fill mapping): Title → Trip title. Resolved `activity_weights` → Trip `activity_weights` (spec 006 FR-017a). `destination_hint` → added as the Trip's first `suggested_destination` row with `ai_reasoning = "Inherited from yearly plan"`. Start month/date, duration → Trip's month/duration fields if those exist; otherwise captured in the Trip's notes. The user can edit any of these in the trip planner before doing further planning.
- **FR-014b** (slot-commit exclusivity): The commit endpoint MUST refuse with `409 {code: "slot_already_committed", committed_option_id: ...}` if any of the option's linked slots already has a different committed option.
- **FR-014c** (uncommit): System MUST expose `POST /api/trip-options/{id}/uncommit` that clears `committed_trip_id` on the option WITHOUT deleting the underlying Trip. The UI surfaces this as a one-click reversal in the commit dialog.
- **FR-014d** (dangling committed trip): If the linked Trip is deleted via the existing trip planner, the application MUST clear `committed_trip_id` on the option as part of the Trip-delete cascade; the option reverts to `shortlisted`. Detection is done at Trip delete time, not via periodic reconciliation.

#### Overview UX

- **FR-015** (timeline): The Year Plan detail page MUST render a horizontal timeline (Jan→Dec) for viewports ≥1000px, with slots positioned by their `start_date` or `(start_month, duration_weeks)` and width proportional to duration.
- **FR-015a** (month grid fallback): For viewports <1000px, the page MUST render a vertical month grid (one row per month) with slots stacked inside their month rows.
- **FR-015b** (slot card): Each slot card MUST show: label, resolved dates/duration, Pending/Shortlisted/Excluded counts (with zero-state), committed Trip link (if any) at top, up to 3 top shortlisted options inline with status chips.
- **FR-015c** (overlap visual): Overlapping slots MUST be visually distinguishable (e.g., vertical offset, border accent) and NOT hidden behind each other.
- **FR-015d** (slot drawer): Clicking a slot MUST open a drawer with the full option list for that slot, grouped by status, with triage buttons per option.
- **FR-015e** (option card detail): Each option card in the drawer MUST show: title, `destination_hint`, `ai_reasoning` (collapsed by default, expandable), any override chips vs the slot baseline, status chip, linked-slots list (if >1), committed-elsewhere badge (if the option is committed on a different slot).

#### Year rollover

- **FR-016** (auto-lock): The system MUST treat a Year Plan as `locked` when the server date is ≥ `{year+1}-01-01` UNLESS `locked_overridden_at` is set AND `locked_overridden_at > server_start_time` (the in-memory timestamp captured on uvicorn startup). Implementation may be computed-at-read-time or materialized; the contract is that writes refuse after year-end except within the override window described above.
- **FR-016a** (unlock): The system MUST expose `POST /api/year-plans/{id}/unlock`, which sets `locked_overridden_at = now()` and allows writes for the remainder of the current server-process lifetime. Server restart (including `uvicorn --reload` on code changes) re-applies the lock because the new `server_start_time` exceeds the persisted `locked_overridden_at`. Closing a browser tab does NOT by itself re-lock — it is the server uptime, not client session, that gates the override. No session cookies, session tables, or JWTs are introduced.
- **FR-016b** (locked-write refusal): When locked, all write endpoints on the Year Plan and its children (slots, options, conversations under the plan) MUST return `403 {code: "year_locked", unlock_url: "/api/year-plans/{id}/unlock"}`. Read endpoints remain available.

#### Data safety (constitution I reinforcement)

- **FR-017**: Schema additions MUST use `ALTER TABLE ADD COLUMN` / `CREATE TABLE IF NOT EXISTS` — no dropping existing tables. The `owner_type` + `owner_id` addition to `conversations` is additive; `trip_id` is retained as a read-only compatibility shadow.
- **FR-017a** (cascade rules): Deleting a Year Plan cascades to: slots, Trip Options (via `slot_trip_options`), conversations + messages scoped to this Year Plan. It does NOT delete linked Trips (their back-link is cleared). Every cascade requires explicit user confirmation in the UI.
- **FR-017b** (slot delete guard): Deleting a slot MUST be refused if any of its Trip Options are `shortlisted` or have `committed_trip_id` set, returning a structured blocker list (same pattern as spec 006 FR-020a). `suggested` and `excluded` options are detached from the deleted slot (deleted if orphaned).
- **FR-017c**: `vacation.db` remains read-only; the yearly chatbot's `search_destinations` calls use the same read path as trip-chat (constitution I).

### Key Entities

- **Year Plan**: A calendar-year container for planning. Holds free-text `overall_intent`, owns slots, owns yearly-scoped conversations, has lifecycle `active` / `locked`. Multiple Year Plans per year allowed (for what-if plans). First-class entity.
- **Slot**: A time window within a Year Plan available for a vacation. Defined by `(start_month, duration_weeks)` and/or `(start_date, end_date)`. Carries baseline `activity_weights`, `climate_hint`, and `constraints`. Owns a many-to-many relationship with Trip Options. Has per-slot `status` (`open` / `locked`).
- **Trip Option**: A candidate trip idea for one or more slots. Has a `title`, a `destination_hint` (freetext or stable `country_code:region_name`), optional field-level overrides against its slot(s)' baselines, AI reasoning, and lifecycle `suggested` / `shortlisted` / `excluded` / `orphaned`. When committed, links to an existing Trip via `committed_trip_id`.
- **Slot–Trip Option Link**: Many-to-many join with ordering. A Trip Option has at least one slot link; losing the last link orphans the option (soft-archive, not delete).
- **Conversation (extended)**: Existing entity extended with `owner_type` (`trip` / `year_plan` / `trip_option`) + `owner_id`. All three scopes share the same table and message model. `trip_id` is retained as a read-only shadow for this feature's backward compatibility window and dropped in a future spec.
- **Trip (existing, extended by spec 006)**: Unchanged by 007 except that it MAY now be the target of a Trip Option's `committed_trip_id`. Trips remain standalone first-class entities — a Trip without a Year Plan link is still a valid Trip.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A new user can create a Year Plan, add 4 slots, write an overall intent, and get AI-suggested Trip Options in at least 3 of the 4 slots within 5 minutes of opening the yearly chatbot for the first time.
- **SC-002**: For a Year Plan with 4 slots and visit history containing ≥2 `not_soon` entries, the yearly chatbot produces ≥3 Trip Option suggestions per slot on the first turn, and 0 of those suggestions target a `not_soon` region — measured by inspecting the `destination_hint` against `vacation.db.region_visits`.
- **SC-003**: 100% of Trip Options entering `shortlisted` or `excluded` come from an explicit user action (HTTP POST from the UI), never from a tool call — same test as spec 002 SC-001 extended to Trip Options.
- **SC-004**: When the user clicks "Plan this trip" on a shortlisted option, the existing trip planner opens with ≥4 pre-filled fields (title, month/duration, activity_weights, and at least one seeded destination or destination_hint) in ≤2 seconds.
- **SC-005**: The unified conversation-owner refactor is verifiable by a data-integrity check: 100% of `conversations` rows have a non-null `(owner_type, owner_id)` pair; for `owner_type = 'trip'` rows, 100% match their (retained) `trip_id`; for `owner_type = 'year_plan'` rows, 100% reference an existing Year Plan.
- **SC-006**: A Year Plan whose `year` is in the past is `locked` on first access in a new session; all write attempts return `403 {code: "year_locked"}`. Zero past-year plans are deleted or silently mutated during rollover.
- **SC-007**: The yearly overview renders 1 Year Plan with 6 slots and 30 Trip Options in ≤500ms on the wide-viewport timeline layout.
- **SC-008**: Cross-slot suggestion dedup works: when the AI calls `suggest_trip_option_for_review` 3 times in one turn with the same `title` targeting 3 different slots, exactly 1 Trip Option record is created and linked to all 3 slots via `slot_trip_options`.
- **SC-009**: The yearly chatbot references the user's existing committed Trips or past visits in its `ai_reasoning` on ≥80% of suggestions, verified by inspecting message content for visit/trip names across a 10-message test conversation.

## Assumptions

- **Domain folder layout (restructure in progress)**: The backend is being restructured so each domain lives in its own package — `backend/app/trips/` (existing trip planning: chat, crud, models, schemas, vacationmap) and `backend/app/golf/` (spec 006: crud, extraction, fetcher, models, schemas, seed_data) are already split. This feature MUST add a new `backend/app/yearly/` (or equivalent) package owning the yearly planner's models, crud, schemas, chat/prompt assembly, and tool definitions. Cross-package references (e.g., Trip Option → Trip via `committed_trip_id`) follow the same pattern spec 006 established: FKs/IDs are plain INTEGER columns on the shared `trips.db`, enforced in application code, not at the DB level. The shared `conversations` table lives where it already does (trips or a neutral `chat` package) but its model is imported by all three domains via the unified `(owner_type, owner_id)` shape (FR-010). No circular imports.
- Single-user, local-app assumptions from constitution VI still apply: no multi-tenant locking on Year Plans, no real-time sync between parallel tabs, "last write wins".
- The yearly chatbot reuses the existing Claude model configuration (constitution: Claude Sonnet 4). No new model selection is introduced. Token cost is expected to be higher than trip-chat because the system prompt now includes all slots + options + multi-plan context; no quota or throttle is introduced (trusted-user stance, same as spec 006).
- The yearly chatbot's tool surface is a superset of trip-chat's read tools plus the single yearly mutating tool (`suggest_trip_option_for_review`). It explicitly does NOT get access to trip-scoped mutating tools (the yearly layer does not manipulate a specific trip's destinations — that's the per-trip chatbot's job).
- The conversation-owner refactor is a one-way migration. Once shipped, the `trip_id` column on `conversations` is read-only shadow data. A follow-up spec will drop it after a verification window.
- Trip Option → Trip commit does NOT copy the option's AI reasoning into the Trip's chat history. The Trip starts with its own clean conversation. The link back to the Year Plan is metadata only; the two conversations remain separate.
- The planned per-trip-option chatbot (mentioned in the feature request as forward-looking) is explicitly out of scope for v1 of 007; only the schema and enum values enabling it are shipped. Building it will be a small additive spec on top of this refactor.
- Year rollover is timezone-lenient: locking uses the server's local date at `{year+1}-01-01 00:00` regardless of the user's timezone. Edge cases where a user is still in the old year when the server flips are acceptable (the unlock button exists).
- Overlapping slots are expected during planning (alternatives, tentative dates). Making them a hard error would force users to delete-and-recreate during normal planning iteration. The warning is sufficient signal.
- Visit-history filtering semantics from spec 005 (3-tier: `never` / `not_soon` / `few_years` / `anytime`) apply identically at the yearly scope. The yearly chatbot calls the same `search_destinations` path with the same filter semantics.
- Existing per-trip chatbot behavior is unchanged by this feature except for the internal refactor (conversation-owner becomes polymorphic; `trip_id` still populated for compatibility). Existing trip conversations load, display, and accept new messages with byte-identical UX.
- The feature does NOT add multi-user sharing, iCal/Google Calendar export, budget optimization across the year, or a "what should my slots even be?" AI tool. Those are explicit non-goals for v1 per the feature request.

## Constitution Check

- **Principle I (Data Safety)**: ✅ All new tables are additive. `conversations` gains two columns; `trip_id` is retained as a read-only shadow for this feature. Year Plan auto-lock is a status flip, never a delete; past plans are never pruned. Cascade deletes on a Year Plan are scoped (children only, linked Trips untouched) and require explicit user confirmation.
- **Principle II (Transparent AI)**: ✅ Yearly chatbot behavior is added to `instructions.md` (new "Yearly planning" section). The system prompt is read fresh each turn from the same files as trip-chat. No hardcoded yearly-specific prompts in Python.
- **Principle III (User Owns Decisions)**: ✅ Exactly one mutating yearly tool (`suggest_trip_option_for_review`); all triage transitions are user-initiated HTTP POSTs; dedup rejects silent re-suggestion of already-triaged options.
- **Principle IV (Stable Identifiers)**: ✅ Trip Options' `destination_hint` that resolves to a VacationMap region uses the existing `country_code:region_name` key. No new cross-DB identifiers introduced.
- **Principle V (Living Documentation)**: ✅ `instructions.md` gains a Yearly planning section; `docs/api-reference.md` and `docs/data-model.md` gain Year Plan / Slot / Trip Option shapes; `CLAUDE.md` gains a Year Plan section; this spec file exists; constitution bump NOT required (no new principle, only reinforcement).
- **Principle VI (Simple by Default)**: ✅ No new major dependency. Reuses existing SQLite, Vue 3 CDN, FastAPI, Anthropic SDK. The polymorphic `owner_type`/`owner_id` pattern is the same shape already used by spec 006 for `entity_images` (enforced in application code, not DB-level FK). The refactor collapses what would otherwise be duplicate conversation tables.

- **Budget / cost optimization across the year**: Not in v1 per the feature request.
- **Calendar export (iCal / Google)**: Not in v1 per the feature request.
- **Multi-user sharing of Year Plans**: Not in v1 per the feature request. Constitution VI (simple-by-default) would need amending first.
